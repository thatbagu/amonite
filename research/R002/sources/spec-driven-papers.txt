SOURCE: Sedeve-Kit: A Specification-Driven Development Framework for Building Distributed Systems
URL: https://arxiv.org/abs/2509.11566 / https://arxiv.org/html/2509.11566v1
Fetched: 2026-07-15

ABSTRACT:
The paper presents a specification-driven development approach addressing the complexity of building distributed systems. The authors state: "Developing distributed systems presents significant challenges, primarily due to the complexity introduced by non-deterministic concurrency and faults."

METHODOLOGY — Five Iterative Stages:
1. Specification Phase: Developers define system behavior and invariants using TLA+
2. Model Checking: "run a model checker on this specification and ensure they pass the model-checking process, guaranteeing correctness"
3. Test Case Generation: TraceGen automatically produces test cases from verified model states
4. Implementation: Developers write code incorporating action anchor macros (AAM) that map to TLA+ specifications
5. Deterministic Testing: D-Player controls test execution using generated traces to validate implementation consistency

HOW SDD DIFFERS FROM INFORMAL TESTING:
The framework addresses a critical gap in traditional approaches. While formal methods verify designs, implementations often diverge during development. Sedeve-Kit closes this by ensuring "the final implementation may deviate from the original design as software evolves and iterates, leading to quality defects."

KEY ADVANTAGES:
- Automated test generation: Eliminates manual test case design
- Exhaustive coverage: Test cases "exhaustively cover the state space corresponding to the design"
- Deterministic validation: D-Player enforces predefined action sequences, enabling reproducible testing of concurrent systems

QUANTITATIVE COMPARISON:
The authors provide one direct comparison: their TLA+ specification for the Raft protocol uses "3038 SLOC TLA+ (including invariants)" versus Verdi's approach requiring "12511 SLOC Coq" for specification and "36925 SLOC" for proofs — demonstrating significantly lower specification overhead while covering more functionality.

FRAMEWORK STAGES DETAIL:
The framework maintains "a strong connection between the abstract design and the concrete implementation through continuous verification," bridging the gap between formal specifications and working code.

SOURCE: Understanding Specification-Driven Code Generation with LLMs: An Empirical Study Design
URL: https://arxiv.org/html/2601.03878v1
Fetched: 2026-07-15
Note: Stage 1 Registered Report, peer reviewed and accepted at SANER 2026.

STUDY OVERVIEW:
This registered report examines LLM-assisted code generation through CURRANTE, a VS Code extension implementing a structured Test-Driven Development workflow. The three-phase approach progresses through specification definition, test suite generation/refinement, and function generation.

KEY RESEARCH QUESTIONS:
RQ1 measures whether the tool effectively generates correct code from user-provided specifications via test cases.
RQ2 evaluates "how well users are able to express their intent producing an effective test suite that captures the functional problem requirements."

METRICS FRAMEWORK:
Effectiveness Metrics:
- PassAll: Binary indicator of all tests passing at task completion
- PassRate: Fraction of passed tests relative to total tests
- TestCoverage: Source code execution coverage during test suite runs
- TestDiversity: Variance among test cases using similarity metrics

Efficiency Metrics:
- TimeToPass: Duration from initial test suite production to all-pass submission
- IterationsToPass: Count of function regenerations until completion

Process Metrics:
- TestEdits: Per-test actions (explain, regenerate, delete)
- SuiteRegenerations: Full test-suite regeneration count
- AdviceTriggers: Requests for AI-generated advice from failures

STUDY DESIGN:
Participants solve medium-difficulty problems from LiveCodeBench dataset with fixed time budgets (30-45 minutes). Between-subjects design assigns each participant one randomly selected problem, using TaskId as a blocking factor to account for task variation.

KEY LIMITATION:
The study acknowledges that "exact reproducibility cannot be guaranteed in human-LLM interaction studies (due to model stochasticity and user behavior)" but mitigates through fixed parameters and multiple independent sessions.

SOURCE: Characterising the Quality of Behaviour Driven Development Specifications
URL: https://pmc.ncbi.nlm.nih.gov/articles/PMC7251619/
Fetched: 2026-07-15

ABSTRACT:
The research investigates what constitutes high-quality Behaviour Driven Development (BDD) specifications. The authors surveyed BDD practitioners to identify and validate quality principles for BDD test suites, which can contain hundreds of scenarios that must correctly specify system behavior both collectively and individually.

METHODOLOGY:
- Literature Review: Analyzed scientific literature and grey literature (BDD Addict Newsletter from 2016-2018, StackOverflow, StackExchange) to identify quality facets
- Survey Design: Created a 9-question survey with Likert-scale responses and open-ended questions
- Sample: 56 respondents from 5 continents (60.7% developers, 51.8% with 6-10 years BDD experience)
- Analysis: Used thematic analysis following Braun and Clarke guidelines for open-ended responses

KEY FINDINGS — Four Quality Principles (each supported by ≥75% of respondents):
1. Conservation of Steps: Reuse existing step phrases to reduce comprehension effort
2. Conservation of Domain Vocabulary: Minimize domain terms to reduce cognitive load
3. Elimination of Technical Vocabulary: Prefer domain terms over technical language for stakeholder readability
4. Conservation of Proper Abstraction: Maintain consistent abstraction levels across scenarios

ADDITIONAL THEMES from practitioners:
- Specifications functioning as readable business documentation
- Regular maintenance and stakeholder review
- Reusable yet focused step definitions

CONCLUSIONS:
While practitioners supported all four principles, they consistently prioritized readability and clarity above strict adherence to reuse principles. The research establishes a foundation for formalizing BDD quality assessment, though operationalizing these principles into measurable metrics requires future investigation.

SOURCE: Formal Modeling and Simulation at Datadog
URL: https://www.datadoghq.com/blog/engineering/formal-modeling-and-simulation/
Fetched: 2026-07-15

MAIN ARGUMENTS FOR FORMAL SPECIFICATION:
The article contends that traditional testing methods alone are insufficient for distributed systems:
"Experience has taught us that it is nearly impossible to intuitively reason about the behavior of a complex distributed system"

The authors argue formal modeling captures design errors that component-level testing misses, as "higher-level design errors are the hardest to find and the most costly to fix."

SPECIFIC BUGS DISCOVERED:
Through TLA+ modeling: When the team added a sequencer to Courier's architecture and re-ran the model checker, it detected a critical race condition. The model revealed that if the sequencer ran infrequently, messages could be sent but not delivered before a receiver gave up after three attempts — a failure mode the informal design review had missed.

Through performance testing: Engineers discovered FoundationDB's range-based sharding created CPU hotspots at the queue tail, causing cascading failures across storage nodes. This required architectural redesign with randomized shard selection.

QUANTITATIVE RESULTS:
The simulation demonstrated graceful degradation:
- In optimistic failure scenarios, Courier matched theoretical best-case linear throughput degradation
- In pessimistic node-loss scenarios, multi-cluster design outperformed single-cluster alternatives
- Chaos testing validated simulation predictions for real implementation behavior

KEY TRADEOFF:
The authors acknowledge formal methods don't validate implementations, only designs, and require ongoing maintenance as systems evolve.
